Container & Dockerfile

Container running as root — no non-root USER defined High
backend/dockerfile

The backend Dockerfile does not define a non-root user. Processes inside the container run as UID 0 (root). If the container is compromised, the attacker gains root-level access to the container filesystem and can potentially escape to the host.

Fix: Add RUN addgroup -S appgroup && adduser -S appuser -G appgroup and USER appuser before the CMD instruction.
Container running as root — no non-root USER defined High
frontend/dockerfile (final stage)

The frontend Dockerfile (both build and final stages) runs as root. The serve package is installed and run globally as root without any user isolation.

Fix: In the final stage add RUN addgroup -S appgroup && adduser -S appuser -G appgroup && chown -R appuser:appgroup /app and USER appuser before CMD.
Flask development server used as production entrypoint Medium
backend/dockerfile:22 — CMD ["python3", "app.py"]

The Dockerfile starts the app via python3 app.py which invokes Flask's built-in development server (app.run()). The dev server is single-threaded, lacks production hardening, and is not recommended for production use.

Fix: Use a production WSGI server. Replace CMD ["python3", "app.py"] with CMD ["gunicorn", "--bind", "0.0.0.0:5001", "--workers", "4", "app:app"] and add gunicorn to requirements.txt.
No HEALTHCHECK defined in backend Dockerfile Low
backend/dockerfile

Without a HEALTHCHECK instruction, Docker and orchestrators (Kubernetes, ECS) cannot determine if the container is healthy, reducing reliability of rolling deployments and auto-restarts.

Fix: Add HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:5001/ || exit 1.
No HEALTHCHECK defined in frontend Dockerfile Low
frontend/dockerfile (final stage)

Same health-check gap as the backend image.

Fix: Add HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:3000/ || exit 1 to the final stage.

Static Analysis (SAST)

Exception details leaked to API clients Medium
backend/app.py:219 — return jsonify({"error": str(e)}), 500

Three API endpoints (get_allcerti, update_certi, issue_certificate) return raw exception messages in HTTP responses. This may expose internal stack traces, database schema details, or file paths to clients, aiding attackers in reconnaissance.

Fix: Log the full exception server-side using Python's logging module and return a generic error message to the client: return jsonify({"error": "An internal error occurred. Please try again."}), 500.
No CSRF protection on state-changing endpoints Medium
backend/app.py — all POST/PUT routes

The Flask app uses session cookies for authentication but has no CSRF token validation. A malicious third-party website could craft cross-origin requests that trigger authenticated actions (e.g., issuing certificates) if a victim's browser sends the session cookie.

Fix: Add Flask-WTF and enable CSRFProtect(app), or set SameSite=Strict on session cookies via app.config['SESSION_COOKIE_SAMESITE'] = 'Strict' and app.config['SESSION_COOKIE_SECURE'] = True.
No input validation on date fields — potential crash via malformed input Medium
backend/app.py:143, 145 — datetime.strptime(dob, '%Y-%m-%d')

datetime.strptime will raise an unhandled ValueError if dob or start_date are malformed strings. This exception propagates uncaught and may expose error details. Similar risk exists in post_addstud.

Fix: Wrap date parsing in a try/except and return a 400 error with a clear message: return jsonify({"error": "Invalid date format. Use YYYY-MM-DD."}), 400.
Missing unique constraints on username and email in database models Medium
backend/models.py:26-27 — Stud.username, Stud.email

The Stud and Admin models define username and email without unique=True constraints. Race conditions or concurrent requests could create duplicate records, potentially bypassing the application-level duplicate check.

Fix: Add unique=True to all identifier columns: username = db.Column(db.String(150), unique=True, nullable=False) and email = db.Column(db.String(255), unique=True, nullable=False).
Sensitive blockchain data logged via console.log Low
frontend/src/components/stucerti.js:51, 63

Student certificate data (ID, grade, program, issuer address) is logged to the browser console, which may be accessible to browser extensions or shared screen recording tools.

Fix: Remove console.log and console.error calls in production builds, or use a conditional: if (process.env.NODE_ENV !== 'production') console.log(...).
alert() used for error display — poor UX and information disclosure Low
frontend/src/components/stucerti.js:66

alert('Error verifying certificate: ' + error.message) exposes raw error messages (which may include contract addresses, revert reasons, or internal details) in a browser popup. alert() is also blocked in some environments.

Fix: Replace with a React state-based error display component that shows a safe, user-friendly message.

Infrastructure as Code

PostgreSQL port exposed directly on host network Medium
docker-compose.yml:29-30 — ports: "5432:5432"

The database port is bound to all host interfaces (0.0.0.0:5432). In any environment beyond a developer laptop (staging, CI, cloud), this allows direct network access to PostgreSQL, bypassing the application layer entirely. Attackers with network access can attempt brute-force or exploit database vulnerabilities directly.

Fix: Remove the ports mapping for the db service. Services in the same Docker Compose network can communicate by service name (e.g., postgres://db:5432/...) without exposing the port externally.

Dependency Vulnerabilities

react-scripts@5.0.1 — known transitive vulnerabilities Medium
frontend/package.json:27 — "react-scripts": "5.0.1"

react-scripts@5.0.1 (released 2022) pulls in multiple dependencies with known vulnerabilities including nth-check (ReDoS, CVE-2021-3803), postcss (line-return parsing, CVE-2023-44270), and loader-utils (prototype pollution). These affect the build toolchain — not the production bundle — but are flagged by npm audit.

Fix: Run npm audit fix to apply safe patches. For breaking changes, consider migrating from react-scripts to Vite (npm create vite@latest) which has a smaller, more actively maintained dependency tree.

Smart Contract

getAllCertificates() — unbounded loop risks gas exhaustion / DoS Medium
blockchain/contracts/CertificateIssuer.sol:92-113

The getAllCertificates() function iterates over the entire studentIds array with no pagination. As certificates accumulate, the gas cost of this call grows linearly and will eventually exceed block gas limits, making the function permanently un-callable on-chain. Off-chain callers (ethers.js view calls) will also time out for very large datasets.

Fix: Add pagination: function getCertificates(uint256 offset, uint256 limit) external view returns (...). Alternatively, rely on event indexing (CertificateIssued events) for off-chain enumeration and remove getAllCertificates().
No emergency pause mechanism in smart contract Low
blockchain/contracts/CertificateIssuer.sol

The contract has no circuit-breaker or pause mechanism. If a critical bug is discovered post-deployment, there is no way to halt certificate issuance while a fix is prepared (since the contract is immutable).

Fix: Implement OpenZeppelin's Pausable pattern: import "@openzeppelin/contracts/security/Pausable.sol"; and add whenNotPaused modifier to issueCertificate.

Secrets Detection

Hardcoded blockchain contract address in frontend source Low
frontend/src/components/stucerti.js:17 — 0xA09d1134F843bBC961CcaEce0a9D84f7e21A2538

The deployed smart contract address is hardcoded in the React component. This is not a secret (contract addresses are public on-chain), but it prevents easy redeployment across environments (testnet vs mainnet) without code changes.

Fix: Move to an environment variable: const contractAddress = process.env.REACT_APP_CONTRACT_ADDRESS; and set it in your .env files (already gitignored).

✅ No hardcoded API keys, passwords, private keys, or credentials detected in any scanned file. All sensitive configuration (SECRET_KEY, DATABASE_URL, POSTGRES_PASSWORD) is correctly loaded from environment variables.
