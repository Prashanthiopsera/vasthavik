Secrets Detection

Scanned with gitleaks v8 — source scan (no-git mode) across 1.23 MB of project files.

Generic API Key / Secret Key exposed on disk Critical
.env : line 3

Rule: generic-api-key. A high-entropy value matching a generic API/secret key pattern was found in the local .env file. Value masked: SECRET_KEY=****.

Risk: If .env is accidentally committed or accessed by an attacker (e.g., via directory traversal in the Flask app), the secret key can be used to forge session cookies, leading to full account takeover.

Fix:
  • Confirm .env is listed in .gitignore (it is — ✓ already done).
  • Rotate the SECRET_KEY value: python3 -c "import secrets; print(secrets.token_hex(32))"
  • Use a secrets manager (AWS Secrets Manager, HashiCorp Vault, or Doppler) for production.
  • Never hardcode secrets in source files.

Static Analysis (SAST)

Scanned with semgrep using p/python and p/javascript rulesets across backend/, frontend/src/, and blockchain/contracts/.

Flask app running with host=0.0.0.0 (public exposure) Warning
semgrep · avoid_app_run_with_bad_host
backend/app.py : line 327

Running Flask with host='0.0.0.0' binds to all network interfaces, exposing the development server publicly. The built-in Flask server is not designed for production traffic and lacks security hardening.

Fix:
  • Use a production WSGI server (gunicorn or uWSGI) instead of app.run().
  • In Docker: CMD ["gunicorn", "-b", "0.0.0.0:5000", "app:app"]
  • Or guard with: if __name__ == '__main__': app.run(host='127.0.0.1', debug=False)

Container & Dockerfile

Scanned with hadolint on backend/dockerfile and frontend/dockerfile. Found 3 issues. Checkov IaC scan FAILED (see Scan Coverage).

DL3018 — Unpinned apk package versions Warning
hadolint · DL3018
frontend/dockerfile : line 7

Using apk add <package> without pinning a version causes non-reproducible builds and can silently pull in vulnerable package versions on rebuild.

Fix: Pin versions: apk add --no-cache nodejs=20.x.x-r0 npm=10.x.x-r0 (check Alpine package repo for exact version).
DL3016 — Unpinned npm package install Warning
hadolint · DL3016
frontend/dockerfile : line 19

Using npm install <package> without a pinned version in Dockerfile. This can cause inconsistent builds.

Fix: Use npm install serve@14.2.4 (or current stable), or use npm ci with a lockfile.
DL3059 — Multiple consecutive RUN instructions Info
hadolint · DL3059
frontend/dockerfile : line 9

Multiple consecutive RUN instructions create extra Docker layers. Consolidate for smaller, faster images.

Fix: Combine RUN instructions with &&: RUN apk add ... && npm install ...

Additional Container Risk (from trivy misconfig)

Container image running as root user High
backend/dockerfile, frontend/dockerfile

Neither Dockerfile creates a non-root user. Running as root inside a container is a container breakout risk — if the process is compromised, the attacker may escalate to host root via kernel exploits.

Fix: Add to backend/dockerfile: 
RUN adduser -D -u 1000 appuser
USER appuser
Add to frontend/dockerfile (after build stage): 
RUN adduser -D -u 1000 appuser
USER appuser
