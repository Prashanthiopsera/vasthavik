🔑 Secrets Detection

Hardcoded Flask Secret Key in Source Code Critical
backend/app.py:11

Flask uses secret_key to sign session cookies with HMAC-SHA1. The key 'vasthavik' is committed to git. Anyone with repository access can forge valid admin session cookies without knowing any user's credentials.

Attack vector (zero skill required): pip install flask-unsign && flask-unsign --sign --cookie "{'user_id': 1}" --secret 'vasthavik'

Fix (5 min):
app.secret_key = os.environ.get('SECRET_KEY') or exit('SECRET_KEY env var required')
Database Credentials Hardcoded in Version Control High
docker-compose.yml:13   backend/app.py:16

PostgreSQL credentials postgres:pgadmin are committed to git. Credential rotation requires a code change and new commit — this antipattern makes credentials effectively unrotatable.

Fix (30 min): Replace with ${DATABASE_URL} in docker-compose.yml and load from a gitignored .env file.

🔍 Static Analysis (SAST)

Zero Authentication on All 9 API Endpoints Critical
backend/app.py — all routes

No authentication middleware, no @login_required decorators, no session validation on any of the 9 Flask API routes. Every endpoint is publicly accessible to any HTTP client.

Exposed operations (unauthenticated):

  • GET /student_all — returns ALL students including passwords
  • PUT /update_certi/<id> — modify any certificate status
  • POST /issue_certificate — issue certificates without authorization
  • POST /post_addstud — create or update any student record
Fix (2 hrs): Add session-based auth decorator to all protected routes. 
from functools import wraps

def require_auth(f):
    @wraps(f)
    def decorated(*args, **kwargs):
        if 'user_id' not in session:
            return jsonify({'error': 'Authentication required'}), 401
        return f(*args, **kwargs)
    return decorated

@app.route('/student_all', methods=['GET'])
@require_auth
def get_all_students(): ...
Plaintext Passwords in Database — Exposed via Unauthenticated API Critical
backend/app.py:33, 42, 77   backend/models.py

All Admin and Student passwords are stored as plaintext strings. Login performs direct string comparison. The GET /student_all endpoint (line 77) returns every student's password field in JSON to any unauthenticated caller.

Fix (1 hr + DB migration): Install bcrypt, hash passwords on create, verify on login, remove password field from API responses.
Predictable Auto-Generated Student Passwords Critical
backend/app.py:57

password = f"{first_name}{last_name}{year_of_birth}" produces passwords like JohnDoe1999. University enrollment rosters are often public or accessible to students, making all accounts immediately compromisable.

Fix (15 min):
import secrets
password = secrets.token_urlsafe(16)
CORS with supports_credentials=True and No Origin Restriction High
backend/app.py:13

CORS(app, supports_credentials=True) with no origins parameter allows any domain to make credentialed cross-origin requests with session cookies — enabling CSRF from any website.

Fix (10 min):
CORS(app, supports_credentials=True, origins=['http://localhost:3000', 'https://yourdomain.com'])
No Request Validation — Malformed Input Crashes Endpoints Medium
backend/app.py:31, 42, 51 (multiple routes)

Routes use request.json['username'] (KeyError crash if missing) and access request.json without checking Content-Type (returns None, causing TypeError). Easy DoS — stack traces are returned to clients revealing internal structure.

Fix (1 hr): Add if not request.is_json: return jsonify({'error': 'JSON required'}), 400 and use request.json.get('key', '') pattern throughout.

📦 Dependency Vulnerabilities (npm audit)

Scanned: frontend/package.json  |  Result: 8 Critical, 35 High, 15 Moderate, 15 Low (146 total packages)

elliptic — Private Key Extraction via Malformed ECDSA Input Critical
GHSA-f7q4-pwc6-w24p

The elliptic library (used by ethers.js) is vulnerable to private key extraction when processing malformed ECDSA signatures. An attacker can extract the private key of any wallet interacting with this frontend — breaking the core blockchain security model.

Fix: Upgrade to ethers@^6 which replaces elliptic with @noble/curves.
pbkdf2, sha.js, cipher-base — Cryptographic Key/Hash Failures Critical

pbkdf2 returns static/zero-filled keys for unimplemented algorithms. sha.js and cipher-base allow hash state rewind via missing type checks. These break cryptographic operations relying on these primitives.

form-data — Unsafe Random Boundary Generation Critical

Uses non-cryptographic random for multipart form boundary generation, enabling predictable boundary attacks.

jspdf — ReDoS and Denial of Service Critical

Malformed PDF input triggers catastrophic regex backtracking, causing application-level DoS.

React Router — XSS via Open Redirects (CVSS 8.0) High
GHSA-2w69-qvjg-hvjx

React Router <= 1.23.1 vulnerable to reflected XSS via untrusted path open redirects. The frontend uses react-router-dom which inherits this vulnerability.

Axios — SSRF and Credential Leakage via Absolute URL High

Axios requests with absolute URLs can leak credentials and enable Server-Side Request Forgery attacks against internal network resources.

serialize-javascript — RCE via Unsafe Serialization High

Vulnerable to Remote Code Execution via RegExp.flags and Date.prototype.toISOString(). Used by webpack build tools (webpack-dev-server, css-minimizer-webpack-plugin).

ws — Denial of Service via HTTP Header Flooding High

The WebSocket library used by ethers.js providers is vulnerable to DoS when handling requests with many HTTP headers.

Fix all: cd frontend && npm audit fix --force or upgrade ethers to v6.x which resolves the majority of these transitively.

host.docker.internal DATABASE_URL Fails on Linux Production High
docker-compose.yml:13

host.docker.internal resolves only on Docker Desktop (macOS/Windows). Linux-based production deployments (EKS, ECS, bare metal) will fail to connect to the database at startup — silent boot failure.

Fix (30 min): Add a db service to docker-compose.yml and use db:5432 as hostname, or add extra_hosts: ["host.docker.internal:host-gateway"] for Linux Docker.
pip install Without --no-cache-dir (Hadolint DL3042) Low
backend/dockerfile:10

Pip cache unnecessarily inflates Docker image size.

Fix (2 min): RUN pip install --no-cache-dir -r requirements.txt

⚓️ Smart Contract Security

issueCertificate() Has No Access Control — Any Ethereum Address Can Issue Certificates Critical
blockchain/contracts/CertificateIssuer.sol:24

function issueCertificate(uint256 _studentId, string memory _grade, string memory _program) public — no onlyOwner, no role check, no modifier. Any Ethereum address can issue fraudulent certificates for any student ID with any grade. Blockchain is immutable — fraudulent certificates cannot be deleted.

Impact: The entire blockchain verification system is permanently compromised upon mainnet deployment. The value proposition of the system (immutable, trustworthy certificates) is eliminated.

Fix (2 hrs — requires contract redeployment): 
import "@openzeppelin/contracts/access/Ownable.sol";

contract CertificateIssuer is Ownable {
    mapping(address => bool) public authorizedIssuers;

    modifier onlyAuthorized() {
        require(authorizedIssuers[msg.sender] || owner() == msg.sender,
                "Not authorized");
        _;
    }

    function issueCertificate(
        uint256 _studentId,
        string memory _grade,
        string memory _program
    ) public onlyAuthorized { ... }

    function addIssuer(address issuer) external onlyOwner {
        authorizedIssuers[issuer] = true;
    }
}
studentIds Array Unbounded Growth — Gas Limit Risk at Scale Medium
blockchain/contracts/CertificateIssuer.sol:13, 43

getAllCertificates() iterates the full array. No duplicate check means the same student can appear multiple times. Gas cost: ~100k at 1,000 certs, ~1M at 10,000 certs (approaching Ethereum block limit). Fraudulent re-issuance silently overwrites legitimate certificates.

Fix (30 min): Add mapping(uint256 => bool) private _studentExists and require uniqueness before pushing.
