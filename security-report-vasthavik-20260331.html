Issues Found

Files scanned: backend/app.py, backend/models.py, backend/init_db.py, backend/dockerfile, frontend/dockerfile, frontend/package.json, frontend/src/components/certificateIssuer.js, frontend/src/components/connect.js, blockchain/contracts/CertificateIssuer.sol, docker-compose.yml

🚨 Critical — Credential Exposure

Hardcoded plaintext admin password in init_db.py Critical — FIXED ✅
backend/init_db.py:25 — password='pkoudi1' in SQL INSERT

The database seeding script inserted an admin record with the raw plaintext password pkoudi1 hardcoded directly in the SQL statement. Additionally, the DB password fallback was hardcoded as pgadmin. Since app.py uses check_password_hash() for login, this would also cause admin login to fail (plaintext ≠ hash), breaking the admin account.

Attack vector: Any person with repository access (or CI pipeline logs) could extract both the admin username and password. If the same credentials were reused elsewhere, this creates a lateral movement risk.

✅ Fixed: Password now hashed with generate_password_hash(). DB_PASS now required via os.environ['DB_PASS'] with no fallback. ADMIN_PASSWORD required via environment variable before seeding.

SAST & API Security

No CSRF protection — session cookies vulnerable to cross-site forgery High — FIXED ✅
backend/app.py — all POST/PUT routes (no SameSite cookie config)

The Flask app uses session-based auth with supports_credentials=True in CORS, but the session cookie had no SameSite attribute. In older browsers this means any cross-origin request automatically includes the session cookie, enabling CSRF attacks. An admin visiting a malicious site could have certificates issued, grades changed, or students added without their consent.

✅ Fixed: Added SESSION_COOKIE_SAMESITE='Lax', SESSION_COOKIE_HTTPONLY=True, and SESSION_COOKIE_SECURE (True in non-development). For full CSRF protection, add Flask-WTF in a future sprint.
IDOR: student ID taken from request body on /request_certificate High — FIXED ✅
backend/app.py:~230 — stud_id = data.get("stud_id")

The certificate request endpoint accepted stud_id from the JSON body rather than from the authenticated session. Any logged-in student could forge a request with another student's ID and create or overwrite certificate requests on their behalf — a classic Insecure Direct Object Reference vulnerability.

✅ Fixed: Changed to stud_id = session.get('user_id') — always derived from the authenticated session, never from client input.
Raw exception messages exposed in HTTP responses Medium — FIXED ✅
backend/app.py — get_allcerti, update_certi, issue_certificate

Three endpoints returned str(e) directly in HTTP responses, potentially leaking database table names, column names, SQL error details, or internal stack information to clients — aiding attackers in reconnaissance.

✅ Fixed: Exceptions now logged server-side with Python's logging module. Generic error messages returned to clients.
No input validation on date fields — crash on malformed input Medium — Open
backend/app.py:~145 — datetime.strptime(dob, '%Y-%m-%d')

datetime.strptime raises an unhandled ValueError if dob or start_date contains an invalid format. This causes a 500 response, and the raw error propagates through the exception handler that has now been fixed — but the validation issue remains.

Fix: Wrap all strptime calls in try/except and return 400 Bad Request with a clear message: return jsonify({"error": "Invalid date format. Use YYYY-MM-DD."}), 400.

Container & Dockerfile

Backend container runs as root — no non-root USER Medium
backend/dockerfile:22 — no USER directive (semgrep: missing-user)

The backend container process runs as UID 0 (root). If the container is compromised via application vulnerability, the attacker has unrestricted access to the container filesystem and may attempt container escape to the host.

Fix: Add before CMD: RUN addgroup -S appgroup && adduser -S appuser -G appgroup then USER appuser.
Frontend container runs as root — no non-root USER Medium
frontend/dockerfile:25 — no USER directive in final stage (semgrep: missing-user)

Same issue as backend. Both the build and final stages of the frontend Dockerfile run as root.

Fix: In the final stage add RUN addgroup -S appgroup && adduser -S appuser -G appgroup && chown -R appuser /app then USER appuser before CMD.
No HEALTHCHECK in backend Dockerfile (CKV_DOCKER_2) Low
backend/dockerfile — missing HEALTHCHECK instruction

Without a HEALTHCHECK, Docker and orchestrators cannot detect an unhealthy container. Dead containers silently receive traffic during rolling updates.

Fix: HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:5001/ || exit 1
No HEALTHCHECK in frontend Dockerfile (CKV_DOCKER_2) Low
frontend/dockerfile — missing HEALTHCHECK in final stage

Same gap as backend. The serve process could crash silently.

Fix: HEALTHCHECK --interval=30s --timeout=5s CMD curl -f http://localhost:3000/ || exit 1

Infrastructure as Code

PostgreSQL port 5432 exposed on host interface Medium
docker-compose.yml — db service: ports: "5432:5432"

The database port is bound to all host interfaces (0.0.0.0:5432). In any environment beyond a developer laptop, this allows direct network access to PostgreSQL, bypassing the application layer. Attackers with network access can brute-force credentials or exploit database vulnerabilities directly.

Fix: Remove the ports block from the db service. Services in the same Compose network communicate by hostname (e.g., postgres://db:5432) without external exposure.

Blockchain Integration

Contract address was placeholder '0xYourContractAddress' — blockchain features broken High — FIXED ✅
frontend/src/components/certificateIssuer.js:12

The blockchain certificate issuance component had the contract address hardcoded as the literal string '0xYourContractAddress'. All on-chain certificate operations would fail silently with a cryptic provider error. The real deployed address was only available as a comment in blockchain/deploy.js.

✅ Fixed: Changed to process.env.REACT_APP_CONTRACT_ADDRESS || '0xd35760e2399360C2Db013eaB657C59Acc10385AE' — environment-configurable with the known deployed address as fallback.
Blockchain-DB dual write with no saga or rollback mechanism Medium
frontend/src/components/certificateIssuer.js + backend/app.py:/issue_certificate

Certificate issuance is a two-step operation: (1) DB write via Flask API, (2) blockchain write via frontend ethers.js. These are completely independent with no coordination. If the MetaMask transaction fails after the DB write succeeds (gas limit, user rejection, network error), the DB record exists but no on-chain proof exists — producing an unverifiable certificate.

Fix: Store blockchain_tx_hash and blockchain_status in the certi table. Mark certificates as pending until the on-chain event (CertificateIssued) is confirmed. Use an event listener or polling to update status.

Secrets Detection

SECRET_KEY detected by gitleaks in .env (false positive — properly gitignored) Informational
.env:3 — SECRET_KEY=cd282dc...

Gitleaks detected the SECRET_KEY value in .env. This is a false positive: .env is listed in .gitignore and will never be committed to the repository. All sensitive config (SECRET_KEY, DATABASE_URL, POSTGRES_PASSWORD) is correctly loaded from environment variables in app.py with mandatory presence validation (raise RuntimeError if missing).

Note: Add a .gitleaksignore file or --no-git exception for .env to suppress this false positive in future scans.
