Executive Summary

The vasthavik project is a university certificate management system comprising a React frontend, a Flask (Python) backend, a PostgreSQL database, and an Ethereum smart contract for certificate issuance. The concept is sound — blockchain-backed credential verification is genuinely valuable. However, the current implementation has the security posture of a hackathon prototype, not a production system handling real academic credentials.

Critical finding: The architecture violates the most basic security principles across every layer: no authentication, no password hashing, hardcoded secrets, and an open smart contract that allows anyone to issue certificates. The blockchain layer, which is the system's core value proposition, is entirely broken from a trust perspective.

What Works Well

What Works Well

🔹 Innovative Architecture: Web + Blockchain Integration

The combination of a traditional web stack with blockchain-backed certificate verification is architecturally sound and genuinely valuable. The concept — using an immutable ledger to verify academic credentials — addresses a real-world problem. The system correctly separates concerns: web layer for CRUD, blockchain for trust anchoring.

🔹 Modern Python Backend with Good ORM Choice

Flask 3.0.0 is a modern, actively maintained version. SQLAlchemy 2.0 with Flask-SQLAlchemy is the correct choice for a Python/PostgreSQL stack. The model definitions are structurally sound (aside from the __tablename__ typo).

🔹 Docker Containerization

Containerizing both frontend and backend with Docker is the right approach for deployment consistency. Multi-service docker-compose setup is appropriate for development. The base image choices (Python 3.11, Node 20-alpine) are modern and appropriate.

🔹 Solidity Version Pragma is Properly Locked

The smart contract uses ^0.8.0 which correctly enables built-in overflow protection (SafeMath is no longer needed). The data model for CertificateInfo (studentId, grade, program, timestamp) is well-structured.

🔹 React SPA with Proper Component Structure

Using React for the frontend with react-router-dom for routing is appropriate. The axios-based API client pattern is standard. jspdf integration for certificate PDF download is a thoughtful UX feature.

Risk Landscape

🔴 Critical Risks

CRIT-1: No Authentication Layer Anywhere

There is no authentication middleware on the Flask backend. All 9 API routes are publicly accessible to any HTTP client without credentials. Admin operations, student data retrieval, and certificate issuance all execute without identity verification. This is not a misconfiguration — there is no authentication code in the codebase at all.

Blast radius: Complete. Any external party with network access to the API can read all data and trigger all state changes.

CRIT-2: Plaintext Password Storage and Transmission

Admin and Student passwords are stored as plaintext VARCHAR columns in PostgreSQL. Login endpoint performs string comparison. The GET /student_all endpoint returns the password field in JSON responses. There is no hashing, no salting, no password policy enforcement.

Blast radius: Complete. A single database read or unauthenticated API call exposes all credentials.

CRIT-3: Open Smart Contract Invalidates Core Value Proposition

The issueCertificate() function is callable by any Ethereum address without restriction. This means the blockchain layer — which is the system's only differentiator from a standard web app — provides no trust guarantees. Any actor can issue fraudulent certificates for any student with any grade, and these cannot be revoked once on-chain.

Blast radius: Existential. The system's purpose (trusted, immutable credentials) is eliminated before it can provide value.

CRIT-4: Hardcoded Secrets in Version Control

The Flask secret key ('vasthavik') is hardcoded in source code. PostgreSQL credentials (postgres:pgadmin) are hardcoded in both app.py and docker-compose.yml. Both are committed to git history, making credential rotation effectively impossible without a full git history rewrite.

CRIT-5: Predictable Credential Generation Algorithm

Student passwords are generated as f"{first_name}{last_name}{year_of_birth}". University enrollment data is often available through institutional directories, social media, or direct enumeration. An attacker with access to student names and birth years can precompute all passwords.

🟡 High Risks

HIGH-1: No Blockchain-Database Consistency Mechanism

The certificate issuance flow (app.py) updates the PostgreSQL database first, then calls the Ethereum smart contract. If the blockchain transaction fails (gas issues, network timeout, nonce collision), the database shows the certificate as issued but no on-chain record exists. There is no retry mechanism, no rollback, no reconciliation process.

Impact: Students may believe they have blockchain-verified certificates that do not actually exist on-chain. Employers verifying on-chain will find no record despite the web portal showing "verified."

HIGH-2: Single Database Instance — No Resilience

The PostgreSQL database has no replica, no connection pooling configuration, no backup strategy visible in the codebase, and no health check in docker-compose. The database is accessed via host.docker.internal which fails on Linux production environments.

HIGH-3: Development Server in Production Docker Container

The frontend Dockerfile uses npm run start (webpack-dev-server) instead of a production build. This means file watching, hot module replacement, and source maps are active in production. The bundle is unoptimized (typically 3-5x larger than production build).

🟠 Medium Risks

MED-1: SQLAlchemy __tablename__ Typo Causes Silent Failures

Three of four models use __table__name (double underscore) instead of __tablename__. SQLAlchemy silently ignores unknown class attributes, so the models fall back to default table naming (class name lowercase). This is a silent discrepancy that will cause query failures when using explicit table references or migrations.

MED-2: No Input Validation Layer

API routes directly access request.json['key'] without checking Content-Type or key existence. Any malformed request crashes the endpoint with a 500 error and a stack trace visible to the client. This enables trivial DoS and information disclosure.

MED-3: Smart Contract studentIds Array Grows Without Bound

getAllCertificates() iterates the full studentIds array. No deduplication check allows the same student to appear multiple times. Gas cost makes this function unusable at scale.

MED-4: No Test Coverage

Zero test files detected across frontend (no .test.js), backend (no test_*.py), and smart contract (no test/*.js). No CI/CD pipeline. Changes are deployed with no automated verification.

Implicit Contracts — Hidden Assumptions

Assumption: Blockchain transaction always succeeds before response

app.py updates the database, then calls the smart contract. The code implicitly assumes both succeed atomically.

"A developer adds a gas price check before the ETH call → The check correctly rejects the transaction → But the DB already committed → Student shows as 'certified' on the web portal but has no on-chain record. Employer verification fails silently."

Assumption: host.docker.internal resolves to the database host

docker-compose.yml hardcodes host.docker.internal as the database hostname.

"DevOps deploys to EKS (Linux). Container starts, Flask imports complete, first API call hits SQLAlchemy → OperationalError: could not translate host name 'host.docker.internal'. Zero requests served after deployment. Appears to be a code bug, not infrastructure."

Assumption: __table__name with double underscore sets the table name

models.py uses __table__name (typo) which SQLAlchemy silently ignores.

"A developer writes a migration referencing 'univ' table. SQLAlchemy actually creates 'univ' (class name), but the model silently uses default naming. Migration targets wrong table. Alembic auto-detect generates spurious migration with full table drop/create."

Assumption: Only authorized admins access the API

All 9 routes are public with no authentication. The system implicitly assumes only the admin UI calls these endpoints.

"Student discovers the API URL from browser DevTools. Calls GET /student_all → receives all 500 student records including passwords. Shares on social media. No audit log exists to detect this."

Behavioral Coupling — Invisibly Connected Modules

Flask app.py ↔ SQLAlchemy models.py

The Flask app imports models directly and uses db.session. The __tablename__ typo in models.py changes the actual table name from what the admin thinks it is. Any query using explicit table names in raw SQL (or Alembic migrations) will reference the wrong table. Changing models.py column types requires a Flask migration (alembic), but no migration infrastructure is set up.

Backend Certificate API ↔ Smart Contract State

app.py updates the DB and the blockchain independently with no distributed transaction. The blockchain txHash is stored in the DB, but there is no verification that the transaction actually confirmed on-chain. A UI showing "Certificate verified on blockchain" could be displaying a txHash for a failed transaction.

Frontend ethers.js ↔ Backend Blockchain State

The React frontend uses ethers.js directly to interact with the contract AND the backend also writes to the contract. There is no single source of truth for blockchain state. A frontend write that the backend doesn't know about (or vice versa) creates a split-brain state.

Dead Paths — Code That May Not Behave as Expected

SQLAlchemy models with __table__name typo

The double-underscore attribute __table__name is not a SQLAlchemy reserved attribute. SQLAlchemy uses the class name (lowercase) as the table name by default. The intended table names 'univ', 'admin', 'stud' may or may not match — depends on whether the class names match. If they do, the code works accidentally; if not, queries silently hit wrong tables.

getAllCertificates() at university scale

The smart contract function returns all certificates by iterating the full studentIds array. At 10,000+ students this function will exceed the Ethereum block gas limit and revert. Any frontend code calling this function will receive an error with no fallback pagination.

Flask session without SECRET_KEY in environment

If the code is updated to use os.environ.get('SECRET_KEY') but the environment variable is not set, the fallback None would cause Flask to refuse to start. Currently the hardcoded secret prevents this failure mode, meaning the "fix" for the secret key issue introduces a startup failure mode if not paired with the .env setup.

Decisions to Make

Decision: Session-based Auth vs JWT for the Flask API

The current codebase has Flask sessions available (app.secret_key). The frontend is a SPA. This creates a tension.

Cost of inaction: Every week without auth is a week the API is publicly exploitable.
Option A: Keep Flask Sessions
  • Lower implementation effort (~2 hrs)
  • Works well with existing Flask infrastructure
  • Requires CORS credentials=True (already set)
  • Cookie-based: works from browser, not from mobile/CLI
Option B: JWT (Flask-JWT-Extended)
  • Stateless, works with mobile/CLI in future
  • More implementation effort (~4 hrs)
  • No server-side session storage needed
  • Better for future microservices expansion

Recommendation: Start with Flask sessions (Option A) to ship auth quickly. Migrate to JWT if/when mobile app or API-as-a-product needs arise.

Decision: Which Ethereum Network for Production?

The smart contract currently deploys to Hardhat local network. Production requires a decision on which network to use.

Option A: Ethereum Mainnet
  • Maximum trust and longevity
  • Gas costs: ~$5-50 per certificate issuance
  • Cannot be undone if fraudulent certificates issued before fix
Option B: Polygon / Optimism L2
  • Gas costs: ~$0.01-0.10 per issuance
  • Still EVM-compatible, ethers.js works unchanged
  • Slightly less "prestigious" than mainnet

Recommendation: Fix the access control issue on testnet first, then deploy to Polygon for cost-effective production use. Never deploy the current open contract to mainnet.
