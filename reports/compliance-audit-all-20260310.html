Executive Summary

Repository: github.com/Prashanthiopsera/vasthavik · Branch: main · 1 commit

This platform handles student PII (name, email, date of birth, academic program) and issues immutable blockchain certificates. Despite recent security hardening — PBKDF2 password hashing, RBAC middleware, secrets moved to environment variables, and smart contract access control — the codebase scores 20 / 100 across all four compliance frameworks.

Decisions to Make

Decision 1: Which secrets management approach?

The current flat .env file approach works for development but is insufficient for production compliance. A decision is needed before deploying to a production environment.

Cost of inaction: Secrets rotate manually (or never), increasing the blast radius of any credential compromise. Fails SOC2 CC9.1 and PCI-DSS Req 3.4.
Option A: HashiCorp Vault

Open-source, self-hosted, full secret lifecycle management. Higher operational overhead. Best for on-prem or multi-cloud.

Option B: Cloud-native (AWS/GCP Secrets Manager)

Managed service, simpler operations, native rotation. Requires cloud provider commitment. Best for cloud-hosted deployments.

Recommendation: If deploying to AWS/GCP, use the native secrets manager (lower ops overhead). For on-prem university servers, Vault or Doppler is preferable.

Decision 2: How to handle ethers@5 CVEs?

The ethers@5.x library has 8 critical CVEs but is the core blockchain interaction library. Migrating to ethers@6.x requires API changes across all blockchain interaction code.

Cost of inaction: 8 critical CVEs remain in production indefinitely, violating PCI-DSS Req 6.3.3 and SOC2 CC7.1. Any external scan will flag these as audit failures.
Option A: Migrate to ethers@6

Resolves all CVEs. Requires refactoring blockchain interaction code in frontend and backend. ~2 days effort.

Option B: Override vulnerable transitive deps

Apply npm overrides for follow-redirects and ws to address most critical CVEs without full migration. Lower effort but may miss some.

Recommendation: Start with Option B immediately (30 min), then plan Option A migration in the next sprint.
