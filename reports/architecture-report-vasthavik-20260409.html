Executive Summary

vasthavik is a blockchain-based academic certificate management system built as a monorepo. It comprises a Flask 3.0 REST API backed by PostgreSQL (AWS RDS), a React 18 SPA, and an Ethereum Solidity smart contract (CertificateIssuer) deployed on Sepolia testnet. The system is containerized and deployed to AWS EKS via a GitOps flow with ArgoCD.

The architecture analysis identified 1 critical, 5 high, and 3 medium issues. The most urgent blocker is a missing /health endpoint in the Flask backend — the Kubernetes probes check this route and will crash-loop every pod on the next deployment. The CI/CD pipeline is also incomplete: it builds Docker images but never pushes them to ECR or updates the GitOps kustomization, so no code changes are ever deployed automatically.

The good news: the auth architecture is well-structured with proper decorator-based enforcement, secrets are managed via Kubernetes Secrets (not hardcoded), and the Solidity contract itself has solid access control with authorized-issuer management. These are strong foundations to build on.

What Works Well

What Works Well

✅ Auth decorator pattern is clean and consistent

The @require_auth and @require_admin decorators in backend/app.py are well-implemented with proper @wraps usage. Critically, stud_id is always derived from the session — never trusted from the client body (see /request_certificate line: stud_id = session.get('user_id')). Passwords are hashed with Werkzeug's PBKDF2-based generate_password_hash and verified with check_password_hash. The safe_model_to_dict helper always strips the password field before returning user data.

✅ Secrets properly managed via Kubernetes Secrets — not hardcoded

SECRET_KEY and DATABASE_URL are injected via secretKeyRef in k8s/base/backend/deployment.yaml. The app raises RuntimeError at startup if these env vars are missing — fast fail, no silent misconfiguration. The .env file is correctly listed in .gitignore.

✅ Solidity contract has strong access control

CertificateIssuer.sol implements a robust authorization model: onlyOwner and onlyAuthorized modifiers, an authorized-issuer registry with revocation (authorizeIssuer / revokeIssuer), ownership transfer with events, and duplicate-certificate prevention via _certificateExists mapping. Events are emitted for all state changes.

✅ Kubernetes pod security hardened

The backend deployment runs as a non-root user (runAsNonRoot: true, runAsUser: 1000), drops all Linux capabilities (capabilities.drop: [ALL]), and prevents privilege escalation. Resource requests and limits are defined. These are best practices often skipped in early-stage projects.

✅ Session cookie security is environment-aware

SESSION_COOKIE_SECURE is set to True in production (FLASK_ENV != 'development') and SESSION_COOKIE_HTTPONLY is always True. SESSION_COOKIE_SAMESITE = 'Lax' provides CSRF protection for same-origin requests. The K8s deployment sets FLASK_ENV=production, so the secure flag is always enabled in prod.

S — Spoofing (2 threats)
Asset: Admin and Student accounts
Credential brute-force via unlimited login attempts

POST /post_univ and /post_stud accept unlimited login attempts with no lockout or rate limiting. An attacker can automate password guessing at network speed.

high Existing: None Recommended: Flask-Limiter 5 req/min + account lockout
Asset: Session integrity across replicas
Session loss causes silent re-authentication bypass

In-memory sessions are lost when the Flask pod restarts. Users are logged out unexpectedly. Adding a second replica causes 50% of requests to get 401 errors. Confused users may be tricked by phishing re-login prompts.

medium Existing: None Recommended: Redis session store (Flask-Session)
T — Tampering (1 threat)
Asset: Certificate validity (DB + blockchain)
Off-chain/on-chain state drift corrupts certificate truth

Certificates can be issued in the DB without a blockchain record, or issued on-chain without a DB entry. There is no reconciliation mechanism. An admin could issue a DB certificate that appears valid in the API but has no on-chain proof. Verifiers get contradictory results depending on which system they query.

high Existing: None Recommended: Atomic issuance + blockchain event listener
R — Repudiation (1 threat)
Asset: Certificate issuance audit trail
No audit log for admin actions beyond DB records

Admin certificate issuance, student record creation, and grade updates are logged to Flask's application logger at INFO level only. There is no dedicated audit log, request ID tracing, or tamper-evident audit trail. An admin could issue then revoke a certificate, and there would be no evidence of the action beyond DB timestamps (which can be altered with direct DB access).

medium Existing: Flask INFO logging Recommended: Structured audit log with request IDs
I — Information Disclosure (1 threat)
Asset: Blockchain deployer private key
PRIVATE_KEY in .env could leak if .gitignore is misconfigured

blockchain/hardhat.config.js reads PRIVATE_KEY from process.env, loaded from the repo root .env. The root .gitignore excludes .env, but the blockchain subdirectory has its own .gitignore that does not explicitly exclude ../.env. If a developer runs git add . from inside blockchain/, the parent .env could be captured.

high Existing: Root .gitignore Recommended: Move PRIVATE_KEY to K8s secret + HSM for prod
D — Denial of Service (3 threats)
Asset: Backend availability
Missing /health endpoint causes immediate CrashLoopBackOff

The Kubernetes livenessProbe and readinessProbe both poll GET /health on port 5001. This route does not exist in backend/app.py. Every deployment will cause pods to fail health checks, be restarted by K8s, and enter CrashLoopBackOff. The backend is completely unavailable after any deploy.

critical Existing: None Recommended: Add /health route with DB ping
Asset: Backend availability
Single replica — any pod restart = 100% downtime

Both backend and frontend are configured with replicas: 1. Regular K8s node rotation, rolling updates, or pod evictions cause full service outage until the new pod starts (~15-30 seconds). No PodDisruptionBudget exists.

high Existing: None Recommended: replicas: 2 + Redis sessions + PodDisruptionBudget
Asset: Blockchain certificate listing
getAllCertificates() hits gas limit at ~500 certificates

getAllCertificates() allocates 4 dynamic arrays and iterates all student IDs in a single call. At ~500 entries, the call approaches Ethereum's block gas limit (~30M) and begins reverting.

medium Existing: None Recommended: Add paginated getter function

Advanced Issues

Architectural concerns that are hard to reverse. Requires design decisions.

High Off-chain / On-chain Certificate State Drift — Trust Model Broken LOW complexity

What's wrong: Certificates are stored in two independent systems — PostgreSQL (via /issue_certificate API) and Ethereum (via MetaMask calling CertificateIssuer.sol directly from the browser). These are completely decoupled operations. An admin can issue a certificate in the DB without creating a blockchain record, or issue on-chain without a DB entry. The blockchain's issueCertificate() and the API's /issue_certificate use different student ID schemas and have no shared transaction boundary.

Who can exploit this: Any admin (internal). No external attack required.

Affected files: backend/app.py:/issue_certificate, blockchain/contracts/CertificateIssuer.sol:issueCertificate(), frontend/src/components/certificateIssuer.js

Reversibility cost: High — requires contract migration or event listener service + data reconciliation job.

Cost of inaction: 3 months: First certificate disputes arise. 6 months: Institutions lose trust in verification. 12 months: The blockchain component provides zero value; entire system is just a DB app.

Recommended fix: Define the DB as authoritative. After DB commit, emit a blockchain transaction asynchronously (backend calls contract via Web3.py). Listen for on-chain CertificateIssued events to confirm. Store tx_hash in the certi table.

Medium getAllCertificates() Gas Limit DoS — Hard Ceiling at ~500 Certs NONE complexity

What's wrong: CertificateIssuer.sol:getAllCertificates() iterates all studentIds in one call, allocating 4 dynamic arrays of the full length. At ~500 entries, this approaches Ethereum's block gas limit (~30M gas) and reverts. The frontend calls this function to display all certificates.

Affected file: blockchain/contracts/CertificateIssuer.sol:getAllCertificates()

Reversibility cost: High — requires contract redeployment and state migration.

Cost of inaction: 6 months: Feature fails silently as certificate count grows. 12 months: Contract must be redeployed; migrating existing data requires admin scripts.
// Add to CertificateIssuer.sol
function getCertificatesPaginated(uint256 offset, uint256 limit)
    public view returns (
        uint256[] memory ids,
        string[] memory grades,
        string[] memory programs,
        address[] memory issuers
    ) {
    uint256 total = studentIds.length;
    if (offset >= total) {
        return (new uint256[](0), new string[](0), new string[](0), new address[](0));
    }
    uint256 end = offset + limit > total ? total : offset + limit;
    uint256 size = end - offset;
    ids = new uint256[](size);
    grades = new string[](size);
    programs = new string[](size);
    issuers = new address[](size);
    for (uint256 i = 0; i < size; i++) {
        uint256 id = studentIds[offset + i];
        Certificate memory cert = certificates[id];
        ids[i] = cert.studentId;
        grades[i] = cert.grade;
        programs[i] = cert.program;
        issuers[i] = cert.issuer;
    }
}

Decisions to Make

Decision: How should user sessions be stored for horizontal scaling?

Currently Flask uses in-memory sessions. This works with 1 replica but breaks with 2+ replicas and loses all sessions on pod restart. A decision is needed before increasing replicas beyond 1.

Cost of inaction: Replicas cannot be increased past 1 without session sharing. Every pod restart logs out all users. Autoscaling is impossible.
Option A: Redis session store (Flask-Session + ElastiCache)

Pros: Minimal code change, sessions survive pod restarts, full session invalidation on logout, existing session lifecycle intact

Cons: Adds Redis/ElastiCache infrastructure dependency, slight latency increase (~1ms per request)

Effort: 1 week

Option B: Signed JWT cookies (stateless)

Pros: No external infrastructure, scales infinitely, fast

Cons: Cannot invalidate tokens server-side (logout doesn't work until expiry), requires JWT refresh logic, significant code refactor

Effort: 2 weeks

Recommendation: Option A — Redis session store. Minimal code change, supports true logout, proven pattern for Flask apps. Add ElastiCache to Terraform in the same sprint as replica increase.

Decision: Which system is the authoritative source of truth for certificates?

Certificates currently exist in both PostgreSQL and Ethereum with no synchronization. Verifiers get different results depending on which system they query. A decision on the authority model is needed before the blockchain component can provide real value.

Cost of inaction: State drift grows with every certificate issued. Verifiers lose trust in the system. The blockchain component becomes unused.
Option A: DB is authoritative, blockchain is audit log

Pros: Practical for day-to-day operations, DB can be queried without RPC, faster UX, blockchain provides tamper-evident record

Cons: Blockchain is secondary; institutions must trust the DB operator

Effort: 2 weeks (event listener service)

Option B: Blockchain is authoritative, DB is cache

Pros: Maximum trust — no central DB operator required for verification

Cons: RPC must always be available, gas costs for every cert, harder UX, requires always-online Ethereum connection

Effort: 4+ weeks (full redesign)

Recommendation: Option A — DB authoritative with blockchain as audit log. The backend calls the contract via Web3.py after DB commit, storing the tx_hash. Verifiers can cross-reference either system, with blockchain providing immutable proof.
