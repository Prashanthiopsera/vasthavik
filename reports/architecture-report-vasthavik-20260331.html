Architecture at a Glance

vasthavik is a monorepo containing three distinct layers: a React SPA frontend, a Flask/PostgreSQL backend API, and an Ethereum smart contract for on-chain certificate issuance. The system allows universities to issue tamper-proof academic certificates verifiable on the blockchain.

What Works Well

✅ Password Security Done Right

Passwords are hashed with Werkzeug's PBKDF2-SHA256 (generate_password_hash / check_password_hash) — industry standard. Passwords are never stored in plaintext in app.py. The secret key is loaded from environment variables with a hard startup failure if missing (raise RuntimeError).

✅ Clean Role-Based Authorization Model

The backend has a clear two-tier access model: @require_auth for authenticated users and @require_admin for admin-only operations. Decorators are applied consistently across all 8 protected endpoints. The role is stored in the session at login and verified on every request.

✅ Smart Contract Security Best Practices

CertificateIssuer.sol demonstrates solid Solidity hygiene: onlyOwner and onlyAuthorized modifiers guard all write operations, ownership transfer emits events, zero address is rejected in authorizeIssuer, duplicate certificate prevention via _certificateExists mapping, and all state changes emit indexed events for off-chain indexing.

✅ Proper Secret Management

All sensitive config (SECRET_KEY, DATABASE_URL, POSTGRES_PASSWORD) is loaded exclusively from environment variables. The .env file is correctly listed in .gitignore. No hardcoded secrets in committed code (the init_db.py plaintext password was a seeding script issue, now fixed).

✅ Multi-Stage Docker Build for Frontend

The frontend Dockerfile uses a proper multi-stage build: Node.js builder stage compiles React, then a minimal Alpine image serves only the built artifacts with serve. This keeps the production image lean and avoids shipping build tooling into production.

Advanced Issues — Architectural Risks

CSRF on admin certificate issuance — no token validation High

SameSite=Lax is a partial mitigation, but top-level navigations still send the cookie. Full CSRF token protection via Flask-WTF is required for POST/PUT endpoints. A malicious site can forge certificate issuance requests if admin is logged in.

backend/app.py:295-325 | Exploitable by: any website admin visits
Blockchain-DB dual write with no saga/rollback High

Certificate issuance is a two-step operation with no coordination: DB write via Flask API + blockchain write via frontend ethers.js. If the MetaMask transaction fails after the DB write, the DB shows "Issued" but no on-chain proof exists. Silent state divergence — unfalsifiable but fraudulent credentials possible.

Cost of inaction: 3mo: Silent divergences. 6mo: Employer verification fails for graduates. 12mo: System untrusted.

Deep Insights

Deep Insights

Configuration Contradictions

CORS Credentials + No CSRF Tokens

supports_credentials=True in Flask-CORS means the browser automatically sends session cookies on cross-origin requests. Combined with no CSRF token validation, this is a textbook CSRF setup. The SameSite=Lax fix reduces but does not eliminate the risk.

ConfigA: supports_credentials=True, ALLOWED_ORIGINS=http://localhost:3000 (backend/app.py:17)
ConfigB: No X-CSRFToken header validation on any POST/PUT endpoint

Flask Dev Server vs Production Needs

Both docker-compose.yml (FLASK_RUN_HOST=0.0.0.0) and the Dockerfile CMD (python3 app.py) run Werkzeug's development server. This is single-threaded and blocks under concurrent load. Should use gunicorn with multiple workers.

Implicit Contracts — Assumptions That Break Under Load

stud_id in /request_certificate matches the authenticated student

Breaks when: Any authenticated student sends stud_id=2 in the request body — creates/overwrites another student's certificate request. FIXED in this scan run.

Blockchain TX and DB write always succeed atomically

Breaks when: MetaMask user rejects the transaction after the DB write completes, or gas estimation fails. DB shows "Issued", blockchain has no record. No rollback, no retry, no notification.

POSTGRES_PASSWORD will be changed from 'CHANGE_ME' before deployment

Breaks when: Developer deploys to staging/production without updating .env. PostgreSQL will start but with a known placeholder password.

MetaMask is always installed and available

Breaks when: Any user without MetaMask visits the app. connect.js silently swallows the error (console.warn only). No user-visible feedback or graceful degradation.
